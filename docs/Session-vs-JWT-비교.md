# Session vs JWT — 동작 원리, 장단점, 적합한 상황

---

## 먼저: 왜 인증 상태를 유지해야 하는가

HTTP는 본질적으로 **무상태(stateless)** 프로토콜입니다. 서버는 요청이 끝나면 그 사람이 누구인지 기억하지 않습니다. 로그인 상태를 유지하려면 "이 요청을 보낸 사람이 이미 인증된 사람이다"를 매 요청마다 증명해야 합니다. 이를 해결하는 두 가지 대표적인 방식이 **Session**과 **JWT**입니다.

---

## 1. Session 기반 인증

### 동작 원리

```
[로그인]
1. 클라이언트 → 서버: ID/PW 전송
2. 서버: 자격 검증 후 메모리(또는 DB)에 세션 생성
   { sessionId: "abc123", userId: 42, role: "user", expires: ... }
3. 서버 → 클라이언트: Set-Cookie: sessionId=abc123 (httpOnly)

[이후 요청]
4. 클라이언트 → 서버: 요청 헤더에 쿠키 자동 포함
5. 서버: sessionId로 저장소 조회 → 유저 확인 → 응답
```

핵심은 **"진짜 데이터는 서버에 있고, 클라이언트는 열쇠(sessionId)만 들고 있다"** 는 점입니다.

### 장점

**즉각적인 세션 무효화가 가능합니다.** 서버 저장소에서 해당 세션을 삭제하면 그 즉시 해당 사용자는 인증이 풀립니다. 강제 로그아웃, 계정 정지, 보안 사고 대응이 실시간으로 가능합니다.

**클라이언트는 민감한 정보를 전혀 갖지 않습니다.** sessionId는 그 자체로는 아무 의미 없는 랜덤 문자열이라, 탈취당해도 즉시 서버에서 무효화할 수 있습니다.

### 단점

**서버가 상태를 들고 있어야 합니다(stateful).** 서버가 여러 대로 늘어나면(수평 확장) 문제가 생깁니다. A 서버에서 로그인했는데 다음 요청이 B 서버로 가면 세션을 찾을 수 없습니다. 이를 해결하려면 별도의 **중앙 세션 저장소**(Redis 등)가 필요합니다.

**모든 요청마다 저장소 조회가 발생합니다.** 트래픽이 많아질수록 저장소가 병목이 됩니다.

## 2. JWT (JSON Web Token) 기반 인증

### 동작 원리

```
[로그인]
1. 클라이언트 → 서버: ID/PW 전송
2. 서버: 자격 검증 후 JWT 생성
   Header.Payload.Signature
   Payload = { userId: 42, role: "user", exp: 1700000000 }
   Signature = HMAC-SHA256(Header + Payload, 서버_비밀키)
3. 서버 → 클라이언트: JWT 발급

[이후 요청]
4. 클라이언트 → 서버: Authorization: Bearer <JWT>
5. 서버: Signature 검증 → 유효하면 Payload 신뢰 → 응답
   (DB 조회 없음)
```

핵심은 **"서버가 아무것도 저장하지 않고, 서명 검증만으로 신뢰 여부를 판단한다"** 는 점입니다.

### JWT 구조 상세

```
eyJhbGciOiJIUzI1NiJ9          ← Header  (알고리즘 정보, Base64)
.eyJ1c2VySWQiOjQyfQ            ← Payload (실제 데이터, Base64)
.SflKxwRJSMeKKF2QT4fwpMeJf36  ← Signature (위변조 방지)
```

Payload는 Base64로 인코딩된 것이지 암호화가 아닙니다. 누구나 디코딩해서 내용을 볼 수 있습니다. **민감 정보(비밀번호, 개인정보)를 Payload에 넣으면 안 됩니다.**

### 장점

**서버가 무상태(stateless)입니다.** 서버 어디에도 저장하지 않으므로 서버를 몇 대로 늘려도 동일하게 동작합니다. 수평 확장이 자연스럽습니다.

**마이크로서비스 환경에 적합합니다.** 여러 서비스가 같은 비밀키(또는 공개키)로 토큰을 검증할 수 있어, 서비스 간 인증 전파가 쉽습니다.

### 단점

**발급한 토큰을 즉시 무효화할 수 없습니다.** 만료 시간이 되기 전까지는 탈취된 토큰도 유효합니다. 이게 JWT의 가장 치명적인 약점입니다.

이를 완화하는 방법이 바로 **Access Token + Refresh Token 분리**입니다.

```
Access Token  - 수명 짧게 (5~15분), 실제 API 요청에 사용
Refresh Token - 수명 길게 (7~30일), AT 재발급 전용

→ AT가 탈취되어도 최대 15분만 유효
→ RT는 httpOnly 쿠키에 저장 + 서버에서 관리
   (RT를 서버에서 관리하면 강제 무효화 가능)
```

실무에서 JWT를 쓸 때 이 구조를 사용하지 않으면 보안상 심각한 문제가 됩니다.

## 3. 핵심 차이 비교

| 항목           | Session          | JWT                              |
| -------------- | ---------------- | -------------------------------- |
| 상태 저장 위치 | 서버 (DB/Redis)  | 클라이언트                       |
| 서버 확장성    | 공유 저장소 필요 | 자유로움                         |
| 즉각 무효화    | 가능             | 기본 불가 (RT 관리 시 부분 가능) |
| 요청당 DB 조회 | 필요             | 불필요                           |
| 토큰 크기      | 작음 (ID만)      | 큼 (Payload 포함)                |
| 구현 복잡도    | 단순             | AT/RT 구조 시 복잡               |

---

## 4. 어떤 상황에 무엇을 선택하는가

**Session이 더 나은 경우:**

- 보안이 최우선인 서비스 (금융, 의료, 관리자 페이지)
- 즉각적인 세션 무효화가 필수인 경우 (강제 로그아웃, 이상 거래 감지)
- 단일 서버 또는 소규모 서비스
- 전통적인 SSR 웹 앱 (Next.js SSR, Rails 등)

**JWT가 더 나은 경우:**

- 마이크로서비스 아키텍처 (여러 서비스가 독립적으로 토큰 검증)
- 수평 확장이 중요한 대용량 서비스
- 모바일 앱 + 웹 앱이 같은 API를 쓰는 경우
- 서비스 간 위임 인증이 필요한 경우

## 5. React SPA 관점에서의 현실적인 문제

React SPA에서 JWT를 쓸 때 자주 빠지는 함정이 있습니다.

```
흔한 실수: localStorage에 AT 저장
→ XSS 공격 한 번이면 토큰 전체 탈취 가능
→ 해결책: AT는 메모리(변수)에, RT는 httpOnly 쿠키에

흔한 실수: 토큰 만료 처리 안 함
→ 401 응답 시 자동 재발급 로직 없으면 UX 파괴
→ 해결책: axios interceptor 또는 fetch wrapper로 처리

흔한 실수: 단일 토큰 구조 (AT만 사용)
→ 만료 시간을 길게 주면 탈취 위험, 짧게 주면 UX 불편
→ 해결책: AT + RT 분리 구조 필수
```

이 프로젝트의 3단계(`feat/jwt-basic`)에서 이 구조를 직접 구현하면서 체감하게 됩니다.

---

**다음으로 이어질 주제는 Token 저장 전략(`localStorage` vs `httpOnly Cookie`)** 입니다. 위에서 언급한 저장 위치 문제를 보안 관점에서 더 깊이 파고드는 내용입니다.
